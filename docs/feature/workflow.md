# Feature 开发工作流 (SDD Workflow)

> 适用范围：所有 Feature 开发必须遵循此闭环工作流

## 总览

```
┌─────────────────────────────────────────────────────────────┐
│                    Feature 闭环工作流                         │
│                                                             │
│  ┌─────┐   ┌─────┐   ┌─────┐   ┌──────┐   ┌─────┐        │
│  │ PRD │──▶│ BED │──▶│ FED │──▶│ TEST │──▶│ 回归 │        │
│  └──┬──┘   └──┬──┘   └──┬──┘   └──┬───┘   └──┬──┘        │
│     │         │         │         │           │            │
│     │    ┌────┴────┐    │    ┌────┴────┐      │            │
│     │    │ 后端实现  │    │    │ 功能测试 │      │            │
│     │    │ + 单测   │    │    │ + 回归   │      │            │
│     │    └─────────┘    │    └─────────┘      │            │
│     │                   │                      │            │
│     │         ┌─────────┴──────────┐           │            │
│     │         │ 前端实现 + 联调     │           │            │
│     │         └────────────────────┘           │            │
│     │                                          │            │
│     └──────────── 反馈闭环 ◀──────────────────┘            │
│           (需求更新/补充/修正)                                 │
└─────────────────────────────────────────────────────────────┘
```

## 阶段一：PRD（产品需求）

### 输入
- 业务需求（来自产品/业务方）
- 用户反馈
- 技术债务

### 活动
1. **明确业务目标**：这个 Feature 要解决什么问题
2. **定义用户场景**：谁在什么场景下做什么
3. **圈定范围**：包含/不包含什么
4. **制定规则**：业务规则、限制、边界
5. **定义验收标准**：怎样算完成

### 输出
- `PRD.md` 文档（基于模板填写）
- 明确的需求边界

### 评审
- [ ] 业务目标清晰
- [ ] 用户场景完整
- [ ] 需求边界明确
- [ ] 验收标准可验证
- [ ] 无遗漏的异常场景

### Agent 协作
```
当 PRD 完成后：
1. 读取 PRD.md → planner agent → 生成实现计划
2. 读取 PRD.md → architect agent → 评估架构影响
3. 读取 PRD.md → 识别涉及的全局规范
```

---

## 阶段二：BED（后端设计）

### 前置条件
- PRD 已评审通过

### 活动
1. **接口设计**：定义 RESTful API 端点
2. **数据结构**：设计数据库表/Schema
3. **服务逻辑**：核心业务处理流程
4. **约束定义**：性能、安全、限流等
5. **单元测试计划**：列出需要覆盖的测试场景

### 输出
- `BED.md` 文档
- API 接口定义
- 数据模型设计
- 测试用例列表

### 评审
- [ ] API 符合 RESTful 规范
- [ ] 数据模型合理（遵循数据库规范）
- [ ] 安全措施到位
- [ ] 性能约束明确
- [ ] 测试场景覆盖完整

### Agent 协作
```
当 BED 完成后：
1. 读取 BED.md → tdd-guide agent → 生成后端测试
2. 读取 BED.md → security-reviewer agent → 安全评审
3. 读取 BED.md → 实现后端代码
```

---

## 阶段三：FED（前端设计）

### 前置条件
- PRD 已评审通过
- BED 已完成（API 接口已定义）

### 活动
1. **页面结构**：组件树和布局
2. **交互设计**：用户操作和系统响应
3. **状态流转**：数据如何在组件间流动
4. **联动逻辑**：组件间/页面间的联动
5. **实现关注点**：性能优化、错误处理

### 输出
- `FED.md` 文档
- 组件设计
- 状态设计
- 交互流程

### 评审
- [ ] 组件设计合理（遵循组件模式规范）
- [ ] 状态管理清晰（遵循状态管理规范）
- [ ] 交互流程完整
- [ ] 错误处理覆盖
- [ ] 响应式/可访问性考虑

### Agent 协作
```
当 FED 完成后：
1. 读取 FED.md → tdd-guide agent → 生成组件测试
2. 读取 FED.md → code-reviewer agent → 代码评审
3. 读取 FED.md → 实现前端代码
```

---

## 阶段四：TEST（测试验证）

### 前置条件
- BED 实现完成 + 单元测试通过
- FED 实现完成 + 组件测试通过

### 活动
1. **功能验收**：逐条验证 PRD 中的验收标准
2. **单元测试审查**：确认覆盖率达标
3. **集成测试**：前后端联调验证
4. **回归评估**：
   - 评估当前 Feature 是否影响已有功能
   - 如有影响，将相关场景加入回归用例库
   - 执行回归测试
5. **回归用例更新**：如有新增回归用例，更新回归用例库

### 输出
- `TEST.md` 文档
- 测试执行结果
- 回归测试结果（如适用）
- 更新的回归用例库（如适用）

### 评审
- [ ] 所有验收标准已通过
- [ ] 单元测试覆盖率 ≥ 80%
- [ ] 集成测试通过
- [ ] 无阻塞性 Bug
- [ ] 回归测试通过（如执行）
- [ ] 回归用例库已更新（如适用）

### Agent 协作
```
当 TEST 阶段：
1. 读取 TEST.md + PRD.md → e2e-runner agent → 执行 E2E 测试
2. 读取 TEST.md → tdd-guide agent → 补充遗漏测试
3. 读取 TEST.md → 评估回归影响范围
```

---

## 闭环：反馈与迭代

### 触发条件
- 验收测试发现需求遗漏
- 实现过程中发现需求不合理
- 技术约束导致需求变更

### 反馈流程
```
发现需求问题
  │
  ├── 补充 PRD（小变更）
  │   └─ 更新 PRD.md → 通知 BED/FED 同步更新
  │
  ├── 修订 PRD（大变更）
  │   └─ 重新评审 PRD → 重新走 BED → FED → TEST 流程
  │
  └── 紧急修复
      └─ 记录变更原因 → 快速修改 → 补充测试
```

## 全局规范引用

在 Feature 开发过程中，应始终参考以下全局规范：

| 规范 | 参考时机 |
|------|----------|
| 编码标准 | 编写任何代码时 |
| 架构原则 | 设计模块结构时 |
| API 规范 | 设计后端接口时 |
| 测试标准 | 编写任何测试时 |
| 安全指南 | 涉及认证/授权/输入时 |
| Git 工作流 | 分支/提交/PR 时 |
| 错误处理 | 设计异常场景时 |
| 状态管理 | 设计前端状态时 |
| 组件模式 | 设计前端组件时 |
| 数据库规范 | 设计数据模型时 |
| 评审清单 | 提交代码评审时 |
