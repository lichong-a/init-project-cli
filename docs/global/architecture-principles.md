# 架构原则 (Architecture Principles)

> 适用范围：全局规范 | 稳定规则 | 跨 Feature 复用

## 核心理念

- **关注点分离**：每个模块只做一件事
- **依赖倒置**：依赖抽象，不依赖具体实现
- **单向数据流**：数据流向可预测、可追踪
- **高内聚低耦合**：模块内部紧密相关，模块之间松散连接

## 项目分层架构

```
src/
├── app/                    # 应用入口与全局配置
├── modules/                # 按业务域划分的模块
│   └── <domain>/
│       ├── handlers/       # 请求处理层（Controller/Handler/Action）
│       ├── services/       # 业务逻辑层
│       ├── repositories/   # 数据访问层（DAO/Repository）
│       ├── models/         # 数据模型/实体
│       └── types/          # 类型定义（如适用）
├── shared/                 # 跨模块共享
│   ├── utils/              # 通用工具
│   ├── types/              # 通用类型
│   └── constants/          # 通用常量
└── infrastructure/         # 基础设施
    ├── http/               # HTTP 客户端/服务端配置
    ├── storage/            # 存储抽象
    └── logger/             # 日志抽象
```

## 模块边界规则

```
允许的依赖方向（从上到下）：
app → modules → shared → infrastructure

modules 之间：禁止直接引用，通过事件/消息总线/接口通信
shared → infrastructure：允许
infrastructure：不依赖任何上层模块
```

## 模块通信模式

| 模式 | 适用场景 | 说明 |
|------|----------|------|
| **函数调用/接口** | 同层模块间 | 通过定义良好的接口解耦 |
| **事件/消息** | 跨模块异步通知 | 发布-订阅模式 |
| **共享状态** | 状态需要多处读取 | 集中式状态管理 |
| **依赖注入** | 服务间协作 | 通过 DI 容器或构造函数注入 |

## 错误边界

每个模块应有自己的错误类型层次：

```
AppError（基础）
├── NetworkError（网络层）
├── ValidationError（验证层）
├── BusinessError（业务层）
├── AuthorizationError（权限层）
└── <Module>Error（模块特定）
```

## 架构评审清单

- [ ] 模块边界清晰，无循环依赖
- [ ] 数据流向单一可追踪
- [ ] 错误处理有明确边界
- [ ] 新增模块符合分层架构
- [ ] 公共逻辑抽取到 shared
- [ ] 无跨模块直接引用
- [ ] 依赖方向正确（不违反分层）
