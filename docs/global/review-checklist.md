# 评审清单 (Review Checklist)

> 适用范围：全局规范 | 稳定规则 | 跨 Feature 复用

## 代码评审维度

### 功能正确性

- [ ] 实现符合 PRD 需求描述
- [ ] 边界条件已处理（空值、异常、极限值）
- [ ] 业务规则正确实现
- [ ] 无遗漏的功能点

### 代码质量

- [ ] 函数 <50 行，文件 <800 行
- [ ] 命名清晰，自解释
- [ ] 无重复代码（DRY）
- [ ] 无深层嵌套（≤4 层）
- [ ] 无魔法数字/字符串
- [ ] 使用不可变模式
- [ ] TypeScript 类型完整

### 架构合规

- [ ] 符合项目分层架构
- [ ] 模块边界清晰，无循环依赖
- [ ] 公共逻辑抽取到 shared
- [ ] 遵循依赖倒置原则

### 安全合规

- [ ] 无硬编码密钥/密码/Token
- [ ] 用户输入已验证和清理
- [ ] 无 SQL 注入风险
- [ ] 无 XSS 风险
- [ ] 认证/授权已验证
- [ ] 错误消息不泄露内部信息

### 测试合规

- [ ] 单元测试覆盖率 ≥ 80%
- [ ] 核心逻辑覆盖率 ≥ 90%
- [ ] 新增代码有对应测试
- [ ] 测试可独立运行
- [ ] 无跳过的测试

### 性能

- [ ] 无不必要的全量渲染
- [ ] 列表使用虚拟滚动（大数据量）
- [ ] 图片/资源有懒加载
- [ ] 无内存泄漏（定时器/订阅清理）

### 可维护性

- [ ] 代码可读性良好
- [ ] 复杂逻辑有注释说明
- [ ] 无 console.log 调试代码
- [ ] 文件组织合理

## PR 评审流程

```
1. 自审（提交前）
   └─ 阅读 diff，检查上述清单

2. Agent 审（自动）
   └─ code-reviewer agent 审查
   └─ security-reviewer agent 审查（敏感操作时）

3. 人工审（至少 1 人）
   └─ 关注业务逻辑和架构
   └─ 使用 conventional comments:
      - [nit]: 小问题（可选修复）
      - [suggestion]: 建议（推荐采纳）
      - [issue]: 问题（必须修复）
      - [blocker]: 阻塞（必须修复才能合并）

4. 修复 & 确认
   └─ 修复所有 blocker 和 issue
   └─ 确认 CI 通过
```

## Conventional Comments 规范

```markdown
<!-- 严重程度从小到大 -->
[nit] 这个变量名可以更具描述性
[suggestion] 考虑将这段逻辑抽取为一个 composable
[issue] 这个 API 调用缺少错误处理
[blocker] 这里存在 SQL 注入风险，必须修复

<!-- 赞美好代码 -->
[praise] 这个抽象做得很好，很清晰
```

## 评审响应时间

| 角色 | 响应时间 |
|------|----------|
| 提交者自审 | 提交前 |
| Agent 审 | < 1 分钟 |
| 人工审 | < 4 小时 |
| 安全审 | < 8 小时 |

## 评审完成标准

- [ ] 所有 blocker 已修复
- [ ] 所有 issue 已修复
- [ ] suggestion 已评估（采纳或说明理由）
- [ ] CI 全部通过
- [ ] 至少 1 人 Approve
- [ ] 无未解决的讨论
